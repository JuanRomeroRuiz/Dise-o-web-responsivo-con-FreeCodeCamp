CONNECT

El método HTTP CONNECT se utiliza principalmente para establecer un túnel a través de un servidor proxy. Este túnel permite que el cliente establezca una conexión directa con un servidor remoto, utilizando un protocolo diferente al HTTP. En otras palabras, CONNECT actúa como un intermediario que facilita la comunicación entre dos puntos, pero sin participar en la conversación.

Por ejemplo: Un cliente quiere conectarse a un servidor IMAP a través de un proxy HTTP. El cliente envía una solicitud CONNECT al proxy, especificando el host y el puerto del servidor IMAP. El proxy establece un túnel y el cliente inicia una sesión IMAP con el servidor, como si se estuviera conectando directamente.

¿Cuándo usar una petición CONNECT?

  • Correo electrónico: Para acceder a servidores IMAP o POP3 que requieren una conexión directa y cifrada, especialmente cuando se está detrás de un firewall restrictivo.
  • VPN: Para establecer conexiones VPN a través de un servidor proxy.
  • Servicios de streaming: En algunos casos, los servicios de streaming pueden requerir conexiones directas para evitar intermediarios que puedan afectar la calidad de la transmisión.

Sintaxis de una petición CONNECT

CONNECT www.example.com:443 HTTP/1.1
  • CONNECT: Este es el verbo HTTP que indica al servidor que se desea establecer un túnel.
  • host:port: La dirección del servidor al que se quiere conectar, incluyendo el número de puerto. Esta parte es crucial, ya que especifica el destino final de la conexión.
  • HTTP/1.1: La versión del protocolo HTTP utilizada.

Ejemplo

CONNECT server.example.com:80 HTTP/1.1 Host: server.example.com:80 Proxy-Authorization: basic aGVsbG86d29ybGQ=
DELETE

El método HTTP DELETE es una de las herramientas fundamentales en la arquitectura REST para gestionar recursos web. Su función principal es eliminar un recurso específico identificado por una URI (Uniform Resource Identifier). Cuando un cliente envía una solicitud DELETE a un servidor, está solicitando de manera explícita que el recurso al que apunta la URI sea borrado de forma permanente.

¿Cuándo usar el método delete?

  • Eliminar elementos de una colección: Por ejemplo, eliminar un comentario específico de una publicación en un blog.
  • Borrar registros de una base de datos: Eliminar un usuario, un producto o cualquier otro tipo de registro almacenado.
  • Anular suscripciones: Eliminar un usuario de una lista de correo o cancelar una suscripción a un servicio.

Sintaxis de una petición DELETE

DELETE /recursos/123 HTTP/1.1 Host: api.ejemplo.com
  • DELETE: El verbo HTTP que indica la acción a realizar.
  • /recursos/123: La URI que identifica de forma única el recurso a eliminar.
  • Host: El encabezado que especifica el servidor al que se dirige la solicitud.

Respuestas del servidor

  • 200 OK: La eliminación se realizó correctamente.
  • 204 No Content: La eliminación se realizó correctamente, pero no hay contenido en la respuesta.
  • 404 Not found: El recurso solicitado no existe.
  • 401 Unauthorized: El cliente no está autorizado para realizar esta acción.
  • 403 Forbidden: El cliente está autorizado, pero no tiene permisos para eliminar el recurso.
  • 500 Internal Server Error: Ocurrió un error en el servidor al procesar la solicitud.
GET

El método HTTP GET es uno de los más comunes y fundamentales en la arquitectura REST. Su función principal es solicitar una representación de un recurso específico. Cuando un cliente envía una petición GET a un servidor, está pidiendo que se le envíe una copia de ese recurso.

¿Cuándo usar el método GET?

  • Obtener información: Para recuperar datos de un servidor, como por ejemplo, una página web, un documento, una imagen o cualquier otro tipo de contenido.
  • Consultar el estado de un recurso: Para verificar si un recurso existe y obtener información sobre su estado actual.
  • Buscar recursos: Para realizar búsquedas en una colección de recursos y obtener los resultados que coincidan con los criterios de búsqueda.

Sintaxis de una petición GET

GET /recursos/123 HTTP/1.1 Host: api.ejemplo.com
  • GET: El verbo HTTP que indica la acción a realizar.
  • /recursos/123: La URI que identifica de forma única el recurso a recuperar.
  • Host: El encabezado que especifica el servidor al que se dirige la solicitud.

Respuestas del servidor

  • 200 OK: El recurso fue encontrado y se envió al cliente.
  • 304 Not Modified: El recurso no ha sido modificado desde la última vez que fue solicitado.
  • 404 Not Found: El recurso solicitado no existe.
  • 500 Internal Server Error: Ocurrió un error en el servidor al procesar la solicitud.
HEAD

El método HTTP HEAD es un método de petición menos conocido pero muy útil. Su función principal es solicitar únicamente los encabezados de una respuesta HTTP, sin el cuerpo de la respuesta. Esto significa que puedes obtener información valiosa sobre un recurso sin tener que descargar todo su contenido.

¿Cuándo usar HEAD?

  • Validación de cabeceras: Para verificar si un recurso existe, su tamaño, tipo de contenido, fecha de modificación, entre otros detalles, sin descargar el contenido completo.
  • Caché: Para determinar si un recurso ha sido modificado desde la última vez que fue solicitado, lo que permite utilizar una versión en caché si no ha habido cambios.
  • Detección de enlaces rotos: Para verificar si un enlace apunta a un recurso válido sin seguir el enlace.
  • Optimización de solicitudes: Al evitar descargar el cuerpo de la respuesta, las solicitudes HEAD son más rápidas y eficientes.

Sintaxis de una petición HEAD

HEAD /recursos/123 HTTP/1.1 Host: api.ejemplo.com
  • HEAD: El verbo HTTP que indica la acción a realizar.
  • /recursos/123: La URI que identifica de forma única el recurso.
  • Host: El encabezado que especifica el servidor al que se dirige la solicitud.

Respuestas del servidor

  • Content-Type: El tipo de contenido del recurso.
  • Content-Length: El tamaño del recurso en bytes.
  • Last-Modified: La fecha y hora en que el recurso fue modificado por última vez.
  • Location: La URL a la que se ha redireccionado el recurso.
  • ETag: Un identificador único del recurso.
OPTIONS

El método HTTP OPTIONS es una herramienta invaluable para los desarrolladores, especialmente cuando se trabaja con APIs RESTful. Su función principal es solicitar al servidor una lista de los métodos HTTP que están permitidos para un recurso específico. En otras palabras, OPTIONS te permite descubrir las acciones que puedes realizar sobre un recurso antes de intentar ejecutarlas.

¿Cuándo usar OPTIONS?

  • Descubrimiento de API: Cuando estás interactuando con una API que no conoces, OPTIONS te permite determinar qué métodos HTTP están soportados para los diferentes recursos.
  • CORS (Cross-Origin Resource Sharing): En solicitudes CORS previas, OPTIONS se utiliza para verificar si el servidor permite solicitudes desde un origen diferente.
  • Seguridad: Al conocer los métodos permitidos, puedes evitar enviar solicitudes no válidas y mejorar la seguridad de tu aplicación.
  • Depuración: Si estás experimentando problemas al realizar solicitudes a un servidor, OPTIONS puede ayudarte a identificar si el problema se debe a un método HTTP no permitido.

Sintaxis de una petición OPTIONS

OPTIONS /recursos HTTP/1.1 Host: api.ejemplo.com
  • OPTIONS: El verbo HTTP que indica la acción a realizar.
  • /recursos: La URI que identifica el recurso para el cual se solicitan las opciones.
  • Host: El encabezado que especifica el servidor al que se dirige la solicitud.

Respuestas del servidor

  • 200 OK: La solicitud fue exitosa y el servidor incluirá en la cabecera Allow una lista de los métodos HTTP permitidos para el recurso.
  • 405 Method Not Allowed: El método OPTIONS no está permitido para el recurso solicitado.

Ejemplo de una respuesta OPTIONS

HTTP/1.1 200 OK Allow: GET, POST, DELETE Content-Length: 0

En este ejemplo, el servidor indica que los métodos GET, POST y DELETE están permitidos para el recurso solicitado.

PATCH

El método HTTP PATCH es una herramienta fundamental para realizar actualizaciones parciales a un recurso existente. A diferencia de PUT, que reemplaza completamente un recurso, PATCH permite aplicar cambios específicos a partes concretas de un recurso.

¿Cuándo usar PATCH?

  • Actualizaciones parciales: Cuando deseas modificar solo algunos atributos de un recurso, sin tener que enviar toda la información del recurso nuevamente.
  • Optimización de tráfico: Al enviar solo los datos que necesitan ser modificados, PATCH reduce el tamaño de las solicitudes y mejora el rendimiento.
  • Versionado: PATCH puede utilizarse para implementar un sistema de versionado, donde cada cambio se registra como una operación individual.

Sintaxis de una petición PATCH

PATCH /recursos/123 HTTP/1.1 Content-Type: application/json
  • PATCH: El verbo HTTP que indica la acción a realizar.
  • /recursos/123: La URI que identifica de forma única el recurso a modificar.
  • Content-Type: El tipo de contenido del cuerpo de la solicitud, que suele ser JSON para especificar los cambios.

Cuerpo de la solicitud

El cuerpo de una solicitud PATCH contiene un conjunto de instrucciones o un "parche" que describe los cambios que se deben aplicar al recurso. El formato exacto del cuerpo puede variar, pero comúnmente se utiliza JSON para representar los cambios.

Ejemplo de un cuerpo de solicitud PATCH en JSON:

JSON { "nombre": "Nuevo nombre", "edad": 30 }

En este ejemplo, se está solicitando que se cambie el nombre del recurso a "Nuevo nombre" y que se actualice la edad a 30.

Respuestas del Servidor

Las respuestas a una solicitud PATCH son similares a las de otras solicitudes HTTP. Los códigos de estado más comunes son:

  • 200 OK: La modificación se realizó correctamente.
  • 204 No Content: La modificación se realizó correctamente, pero no hay contenido en la respuesta.
  • 400 Bad Request: La solicitud contiene errores de sintaxis.
  • 404 Not Found: El recurso solicitado no existe.
POST

El método HTTP POST es uno de los más utilizados para interactuar con servidores web. Su función principal es enviar datos a un servidor para crear un nuevo recurso. Es decir, cuando deseas agregar nueva información a un sistema, POST es el método adecuado.

Cuando usar POST

  • Crear nuevos recursos: Para agregar nuevos elementos a una colección, como crear un nuevo usuario, un nuevo producto o una nueva publicación.
  • Enviar datos de formularios: Para enviar los datos de un formulario HTML a un servidor.
  • Subir archivos: Para enviar archivos al servidor, como imágenes, documentos o videos.

Sintaxis de una petición POST

POST /recursos HTTP/1.1 Host: api.ejemplo.com Content-Type: application/json
  • POST: El verbo HTTP que indica la acción a realizar.
  • /recursos: La URI que identifica la colección de recursos donde se creará el nuevo elemento.
  • Host: El encabezado que especifica el servidor al que se dirige la solicitud.
  • Content-Type: El tipo de contenido del cuerpo de la solicitud, que suele ser JSON, XML o multipart/form-data para enviar archivos.

Cuerpo de la solicitud

El cuerpo de una solicitud POST contiene los datos que se utilizarán para crear el nuevo recurso. El formato del cuerpo depende del tipo de contenido especificado en el encabezado Content-Type.

Ejemplo de un cuerpo de solicitud POST en JSON:

JSON { "nombre": "Nuevo usuario", "email": "nuevo@ejemplo.com" }

Respuestas del Servidor

  • 201 Created:El recurso se ha creado correctamente y la ubicación del nuevo recurso se incluye en la cabecera Location.
  • 200 OK: El recurso se ha creado correctamente, pero la ubicación del nuevo recurso no se incluye en la cabecera Location.
  • 400 Bad Request: La solicitud contiene errores de sintaxis.
  • 403 Forbidden: No tienes autorización para crear el recurso.
PUT

El método HTTP PUT se utiliza principalmente para crear o actualizar completamente un recurso existente. Esto significa que cuando envías una solicitud PUT a un servidor, estás reemplazando la representación actual del recurso con los datos proporcionados en el cuerpo de la solicitud.

¿Cuándo usar PUT?

  • Crear un recurso: Si conoces la ubicación exacta donde deseas crear el recurso, puedes utilizar PUT en lugar de POST.
  • Actualizar un recurso completo: Cuando deseas reemplazar todos los datos de un recurso existente con un nuevo conjunto de datos.

Sintaxis de una petición PUT

PUT /recursos/123 HTTP/1.1 Content-Type: application/json
  • PUT: El verbo HTTP que indica la acción a realizar.
  • /recursos/123: La URI que identifica de forma única el recurso a actualizar.
  • Content-Type: El tipo de contenido del cuerpo de la solicitud, que suele ser JSON para especificar los nuevos datos del recurso.

Cuerpo de la solicitud

El cuerpo de una solicitud PUT contiene una representación completa del recurso tal como deseas que quede después de la actualización.

Ejemplo de un cuerpo de solicitud PUT en JSON:

JSON { "nombre": "Nuevo nombre", "edad": 30, "ciudad": "Nueva ciudad" }

En este ejemplo, se está solicitando que se actualicen todos los datos del recurso con los valores proporcionados en el JSON.

Respuestas del Servidor

  • 200 OK: La actualización se realizó correctamente.
  • 204 No Content: La actualización se realizó correctamente, pero no hay contenido en la respuesta.
  • 400 Bad Request: La solicitud contiene errores de sintaxis.
  • 404 Not Found: El recurso solicitado no existe.
TRACE

El método HTTP TRACE es un método de diagnóstico utilizado principalmente para realizar pruebas en un servidor web. Cuando se envía una solicitud TRACE a un servidor, este debe devolver una respuesta que contenga una copia exacta de la solicitud recibida. Es decir, el servidor "hace eco" de la petición.

¿Cuándo Usar TRACE?

  • Depuración: Se utiliza para verificar la configuración de un servidor, identificar problemas de enrutamiento o comprobar si un proxy está modificando la solicitud.
  • Pruebas de seguridad: Puede ser útil para detectar vulnerabilidades como el ataque de reflexión XSS.

Sintaxis de una Petición TRACE

TRACE /ruta/del/recurso HTTP/1.1
  • TRACE: El verbo HTTP que indica la acción a realizar.
  • /ruta/del/recurso: La URI que identifica el recurso.

Ejemplo de una respuesta TRACE:

HTTP/1.1 200 OK Content-Type: message/http TRACE /ruta/del/recurso HTTP/1.1 Host: ejemplo.com ... (resto de la solicitud)
